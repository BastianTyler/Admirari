Getting Started with Git & GitHub


This page introduces Git and Version Control Software (VCS). We'll start with the basics of VCS, then install Git and set up a GitHub account. After exploring the interface and key terms, we'll install GitHub Desktop for a visual workflow. Finally, we'll practice core Git operations like creating and cloning repositories, making commits, pushing changes, branching, and merging with pull requests.

Written by: Dylan Martin

What is VCS, Git, GitHub, and GitHub Desktop?

Glossary:

  • Version Control Software (VCS) - Helps track and manage changes to files over time. It allows multiple users to collaborate, revert to previous versions, and maintain a history of modifications, ensuring stability and efficiency in software development.

  • Git - Git is a free VCS that tracks project history and enables team collaboration. While commonly used via the command line, this tutorial focuses on GitHub Desktop, a user-friendly GUI for easier version control. If you are interested in Git, you can find its documentation here.

  • GitHub - GitHub is a cloud based hosting service for Git projects, colloquially called repositories or repos for short. It allows developers to store, manage, and track changes to code while enabling team collaboration through features like repositories, branches, pull requests, and issues.

  • GitHub Desktop - GitHub Desktop is a graphical user interface (GUI) application that simplifies using Git and GitHub. It provides an intuitive way to manage repositories, commit changes, create branches, and sync code without using the command line. Ideal for beginners, it lowers the barrier to entry for version control while maintaining powerful Git functionality.
    • Note: GitHub Desktop is only available for Apple and Windows devices.

Key Takeaways

Git, a free and widely used VCS, efficiently manages project history and teamwork. GitHub, a cloud-based platform for hosting Git repositories, enhances collaboration with features like branches, pull requests, and issues. For an easier experience, GitHub Desktop provides a user-friendly GUI that simplifies Git operations without requiring command-line knowledge.

Making Your First Repository

The first step to starting any project is to create a repository, or repo for short. While there are a number of ways to accomplish this, we will be creating our repository through GitHub Desktop. For this example, feel free to use any name you would like for your repository.

Going step by step...

  1. In GitHub Desktop, navigate to the File section of the menu bar, shown below. Select "New repository...", or simply press Ctrl+N.


  2. Typically when making a repo, we can give it any name we want. For this example, I'll name it "admirari-github-workshop".


  3. Make sure to check the box seen above under the 'Local Path' section that says "Initialize this repository with a README."

    What is a README? - A README is a plain text Markdown file that is used to provide essential information about a project. It is commonly found in software repositories, containing information about a project. It's used to inform users and developers of the purpose of a project and how to interact with it.
    We will discuss what exactly Markdown is a bit further down.

  4. Click "Create Repository". GitHub desktop should now show you this screen:


  5. Publish the repository to GitHub by pressing the "Publish repository" button located in the main section of GitHub Desktop as seen in the above image. Alternatively you can publish by pressing Ctrl+P.

  6. You should now see this window. Go ahead and click the blue "Publish Repository" button.

Where is My Repo?

If you followed the instructions correctly, you should now have published your repository on GitHub. If you want to view the files you created, or view the repository on GitHub itself, GitHub Desktop has buttons for these actions shown here:


These options are also available in the menu bar under 'Repository'.

    File > Options > Integrations > External Editor

  2. Editing Our README - Once we have our repository open in our code editor, we can start making changes to our README. Go ahead and click on README.md in the explorer menu.


    I don't see a README in my repo!
    You may have forgotten to create a README when creating your repo in a previous step. That's ok! You can simply right click in the explorer menu of VSCode and select "New File." Make sure to name it "README.md".

  3. Now it's time to talk about Markdown
    • What is Markdown?
      Markdown is a markup language used for formatting text. It allows you to add heading, bold and italic text, list, links, and more - without the need for HTML or styling.
    • Basic Markdown Syntax Examples
      • Bold → **bold text**
      • Italic → *italic text*
      • Lists:
        • - Item 1
        • - Item 2
      • Code block:
        • ```
        print("Hello, Markdown!")
        ```

Making Our First Commit

Now that we have made changes to our project, lets commit these changes to our published project on GitHub. Doing this is a relatively simple process through GitHub Desktop. All you need to do is fill out the information in this section in the bottom right. Since the automatically generated field is descriptive enough, I'm just going to go ahead and press "Commit to main". This will push the changes we made to the README to the project as it is hosted on GitHub itself.
Be careful doing this in larger projects!
You may want to create new branch of the repository for any minor changes. We will discuss branches below.

What if someone else made changes to the project?

In order to make sure you have the most recent version of code for a project, make sure to regularly fetch updates from the repository. You can do this in GitHub Desktop by clicking the following button:


It is good practice to fetch the latest updates to a repository before committing changes!

Let's go to our project in IntelliJ and add a for loop. Mine will run three times and print out each number. I'll run it once to see if it works, then go back to GitHub Desktop. I'll commit these changes to the `feature-one` branch. If I am happy with my changes, I can now try and merge it with my `main` branch. There are several options for this.
